## 🔴 CRITICAL — business model correction, not yet implemented

**The Fleet Owner (the client) owns the trucks.** Vehicles are:
- **Owned** outright, or
- **Rented**, or
- **On EMI** — and the EMI installment is a recurring liability that must be tracked (who it's paid to, amount, due date, outstanding loan balance).

**Transporters are brokers/customers, not vehicle owners.** A transporter calls the owner with a load, the owner assigns one of *his own* trucks and drivers, delivers, then bills the transporter. The transporter never owns or supplies vehicles.

**Current bug:** `Vehicle.transporterId` in `Hope/backend/prisma/schema.prisma` and every place that reads it (VehiclesPage.js, the vehicle create form, etc.) treats "transporter" as if it *owns* the vehicle. That's backwards. This needs a schema + UI redesign:
- Add vehicle ownership fields: `ownershipStatus` already exists (`OWNED`/`ATTACHED`/`LEASED` enum) but nothing consumes it meaningfully yet, and there's no EMI tracking at all (no loan amount, EMI amount, due day, lender, tenure, remaining balance).
- Figure out what `transporterId` on Vehicle was even *for* — if it's meant to represent "vehicle is attached/leased from a transporter-owner" (the `ATTACHED`/`LEASED` cases), that's a different relationship than "this vehicle serves this transporter's loads" and the UI conflates them. Needs a real conversation with the client about what ATTACHED/LEASED actually means operationally before rebuilding this.
- New: an EMI/loan model (per vehicle), recurring liability tracking, probably a `VehicleExpense` category for EMI (the schema already has `VehicleExpense` — check what categories exist).

**Do not build more features on top of the vehicle model until this is redesigned.** It's foundational — trip P&L, vehicle profitability reports, and the dashboard all depend on getting vehicle costs right.

## What this project is

**Transit Ledger** — a ledger-first fleet operating system for Indian SME truckers. This is being built as an MSSP-style SaaS: eventually multiple fleet-owner clients, each with their own isolated dashboard/data.

Real business flow (confirmed by client on call):
1. Transporter (broker) calls the owner/dispatcher with a load — pickup, destination, material, weight, freight amount.
2. Owner checks his own truck + driver availability, assigns them.
3. During the trip: driver may need cash advances, diesel money, may receive money directly from transporter/party — all needs capturing.
4. After delivery: driver sends POD photo via WhatsApp → dispatcher forwards to transporter → bills transporter.
5. Transporter pays in full or parts, across cash/UPI/bank/cheque, over time.
6. Driver settlement happens separately: salary, advances, daily bhatta (fixed rate/day while on a trip), incentives, deductions, penalties.

No formal LR (lorry receipt) generation today — all handwritten. A generator is a future nice-to-have, not MVP.

**Target scale:** 50,000 trips/year, 100 trucks, 200 drivers, multiple clients (MSSP) eventually.

**UI direction (explicit client/product-owner requirement):** Light/whitish theme ONLY — no dark theme, no toggle. Clean, simple, "feels like a khatabook, not SAP." Desktop: persistent left sidebar. Mobile: sticky top header + bottom tab bar with a "More" sheet for overflow. Must never require a page refresh to navigate.

## Trip lifecycle (state machine, enforced server-side)

`DRAFT → LOADING → IN_TRANSIT → DELIVERED → POD_RECEIVED → BILLED → SETTLED` (+ `CANCELLED` from most states). Transitions validated in `Hope/backend/src/routes/trips.js` (`STATUS_TRANSITIONS`). Marking DELIVERED auto-accrues each assigned driver's daily bhatta (`dailyExpenseRate × trip duration`) as `TripExpense` rows.

### Commit `9dc7817` — scaling fix (route-scoped loading) + more contract bugs
The app loaded **all 9 endpoints** (dashboard, reference-data, transporters, vehicles, drivers, routes, trips, ledger-entries, payments) on every page load AND after every mutation, with `/trips` embedding every payment/expense/ledger-entry per trip. At scale (thousands of trips) this would be multi-MB payloads on every interaction.

Fixed:
- **Route-scoped loading** (`main.js`): each page fetches only what it needs, cached 15s, force-refreshed only for the current page after a mutation.
- **Slim `/trips` list**: server-side filtering (transporterId/vehicleId/routeId/driverId/status/search/date-range) + real pagination (50/page default, 200 max) instead of client-side filtering over a fully-loaded array.
- **Bulk-aggregated totals**: transporters/drivers list endpoints return `outstanding`/`freightTotal`/`paidTotal`/`tripCount` computed server-side via 2 `groupBy` queries — pages never need to load full trips/payments arrays just to show a number.
- Rewrote `TripDetailPage`/`DriverDetailPage`/`RouteDetailPage`/`TransporterDetailPage` to fetch their record by ID instead of searching a preloaded global list (this exposed that they'd never actually been fixed in the previous pass — still had fictional fields like `route.fromCity`, `driver.advanceBalance`).
- Found and removed a "Record advance" form that posted to a form type `main.js` never handled — dead code. Advances are just `DriverSettlement` rows with `type=ADVANCE`.
- **Vehicles form bug**: the transporter `<select>` was named `transporterSelect` while a separate, never-synced hidden input was named `transporterId` — vehicles never actually got assigned on create. (Now moot once the vehicle-ownership model above gets redesigned, but the mechanical form bug is fixed.)
- **Driver multi-select FormData collision**: the driver checkboxes and the hidden JSON-array aggregator field both had `name="driverIds"` — `FormData`/`Object.fromEntries` collapsed them, so the raw checkbox value (a single ID string) silently clobbered the array, and every trip submitted with a driver assigned failed Zod validation (`Expected array, received string`). Fixed by removing `name` from the checkboxes.

**Verified end-to-end against a live local dev DB** (not just build-checked): created transporter→vehicle→route→driver→trip chain, assigned a driver, recorded a payment, transitioned trip status, tested delete (both success and FK-guarded-failure paths). No console errors, no full-page flashes.

## What's NOT done — pending work, roughly in priority order

1. **Vehicle ownership model redesign** (see 🔴 above) — blocks trip P&L / vehicle profitability being meaningful.
2. **Multi-tenancy / auth** for the MSSP model. Currently `Hope/backend/src/context.js`'s `getOrganization()` just grabs the first Organization row — single-tenant shim. Schema already has `organizationId` on every model (multi-tenant-*shaped*), but there's no login/session/JWT layer resolving which org a request belongs to. This is a real auth build, not a config flag. Decide: one DB with tenant-scoped rows (less ops overhead, what the schema already assumes) vs. separate DB per client (client said this is acceptable — "yes we can create multiple dbs/khatabooks" — but more overhead).
3. **Driver outstanding-salary formula needs a real business-rule pass.** Current `calculateDriverOutstanding` (`Hope/backend/src/services/calculations.js`) sums settlementTotal (ALL settlement types: salary paid, advances given, deductions, penalties, cash collected, allowances — all added together as positive numbers) + tripExpensesPaid + dailyExpenses accrued. This does NOT net "salary paid out" against "salary owed" — it looks more like "total money that has moved" than a true signed balance. I deliberately did not silently redefine this financial formula; it needs the client's explicit sign-off on what "outstanding" should mean per settlement type.
4. **Trip event timeline** — client's explicit ask (driver started, mid-trip driver swap with cash handover, incidents/accidents, payment received) — doesn't exist at all yet. See roadmap below, item #1.
5. **UI rebuild** — light/whitish theme, proper responsive nav — not started. Current UI is functional but was never redesigned for the "khatabook not SAP" direction.
6. **Remaining smaller bugs not yet fixed** (from a manual QA pass the user did on the live Vercel deployment): no `maxlength` on Transporters/Vehicles/Drivers forms (Trips/Routes/Payment forms already have it), some `required`-only client validation still fails silently rather than showing inline errors, mobile hamburger menu has no explicit close button (overlay-tap and nav-away already close it).
7. **LR/bilty generator** — explicitly deferred by client to a future version.

## Feature roadmap (from background research this session — Indian fleet-SaaS competitors + khatabook UX patterns)

Top priorities, ranked, effort-tagged, mapped to client's WhatsApp asks:

1. **Append-only TripEvent log + trip timeline feed** (M) — the client's #1 ask. Foundation: one `TripEvent` table, indexed `(tripId, at)` and `(driverId, at)`. No competitor at khata-tier or enterprise-tier combines an immutable ledger with a WhatsApp-style trip story — this is the identified differentiator.
2. **Multi-driver per trip with swap event** (M) — `TripDriver` join already exists; add swap timestamp + cash-handover amount as a `DRIVER_SWAPPED` event.
3. **Driver cash-in-hand ledger** (S) — derived: advances + diesel cash given − expenses reported − handed back. Live "driver holds ₹X".
4. **Trip expense line-items** (S/M) — category breakdown (diesel/toll/repair/challan/loading) — prerequisite for #3 and for real trip P&L.
5. **Multiple classified payments + allocation** (M) — mostly done (payment types exist); add allocating one payment across multiple open trips.
6. **Driver Khata + outstanding-salary statement** (M) — ties directly into item #3 above (needs the business-rule fix first). Shareable settlement slip via `wa.me` link.
7. **Trip P&L card** (S) — freight − (advances+diesel+bhatta+tolls+commission) = net profit. Needs vehicle-ownership model fixed first for real accuracy (EMI/rental cost should factor in).
8. **Receivable aging + "You'll Get/You'll Give" dashboard header** (S).
9. **wa.me collection reminders + ledger share slips** (S) — no paid WhatsApp Business API needed, just prefilled `wa.me` links.
10. **POD attach + status workflow** (S/M) — mostly exists server-side (`podImageUrl`/`podNotes`/`podReceivedDate`), UI now wired this session via `CardComponents.createPodForm`.
11. **Incident/accident event** (S) — once #1 (timeline) exists, this is cheap.
12. **Document expiry alerts** (S) — insurance/permit/PUC/fitness/RC + driver license. Schema already has the date fields on Vehicle; nothing surfaces them yet.

**Explicitly do NOT build yet:** GPS/telematics/FASTag integration, e-way bill/GST integration, a return-load marketplace, a dedicated maintenance/job-card module, shop/vendor khata (payables) — all named as later-phase or out-of-scope by the client's requirements doc.
